# claude-fable-5_2026-06-11_0712_39

Fail-closed means: when the system is unconfigured, misconfigured, or something goes wrong, it defaults to the *protective* behavior (redaction happens) rather than the *permissive* one (content flows through raw). Concretely in this plan:

- **Absent config → redaction on.** If `secretsPolicy` isn't in the shared config file at all — which is every existing install after upgrade — the default is `redact` mode, not `off`. Users opt *out*, not in. That's your stated goal ("automatically, by default").
- **Invalid config → rejected, not ignored.** Unknown keys or a bad `mode` value fail config validation (same posture as your existing `featureFlags` handling), instead of silently loading a policy that doesn't do what the user thinks.
- **One choke point → new surfaces can't forget.** Redacting at parse time means every downstream consumer — twins, recordings, snippets, `status.json`, anything added later — inherits redaction automatically. The alternative (enforcing separately at each write site) is fail-*open*: a future feature that reads events and writes them somewhere new would leak unless someone remembered to wire in the redactor.
- **Detector error → don't pass content through raw.** If the redaction transform itself throws on an event, the fail-closed move is to drop/quarantine that event (with an audit log entry), not ship it unredacted. I'd carry that into implementation.
- **Ambiguity → over-redact.** A false positive mangles a transcript that's recoverable from the provider source file; a false negative in a committed recording is unrecoverable.

The term comes from your own [[dev.security-baseline]] invariant #7: "Unconfigured sensitive surfaces MUST fail closed rather than starting with insecure defaults" — this design extends that posture to content redaction.

I'll also drop retroactive cleanup from the note since you don't need it:

In short: fail-closed = "when in doubt, redact" applied at every layer — defaults (absent config means redaction on), validation (bad config is rejected, not guessed at), architecture (one parse-time choke point so nothing downstream can bypass it), and error handling (a failing detector drops the event rather than leaking it).

**What changed**
